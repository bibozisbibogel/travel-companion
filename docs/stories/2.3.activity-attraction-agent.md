## Story
**As a** traveler,
**I want** the system to suggest relevant activities and attractions,
**so that** I can discover interesting things to do at my destination.

## Acceptance Criteria
1. Activity Agent with TripAdvisor and Viator API integrations
2. Activity search based on destination, travel dates, and interest categories
3. Activity data model with descriptions, pricing, duration, and booking information
4. Activity categorization (cultural, adventure, family-friendly, etc.)
5. Activity filtering by budget, duration, and user preferences
6. GetYourGuide API integration as additional data source

**Activity Data Model Requirements**:
```python
# activity_options table schema (already exists)
activity_id: UUID - Primary identifier
trip_id: UUID - Foreign key to Trip
external_id: VARCHAR(255) - API provider's ID
name: VARCHAR(255) - Activity name
description: TEXT - Activity description
category: activity_category - Enum: cultural, adventure, food, entertainment, nature, shopping, relaxation, nightlife
location: POINT - PostGIS geographic coordinates
duration_minutes: INTEGER - Activity duration
price: DECIMAL(10,2) - Activity price
currency: CHAR(3) - Currency code (default USD)
rating: DECIMAL(2,1) - Activity rating (1-5 scale)
booking_url: TEXT - Booking link
created_at: TIMESTAMP - Record creation time
```

### Coding Standards and Testing Requirements
[Source: architecture/coding-standards.md, architecture/test-strategy-and-standards.md]
**Critical Coding Standards**:
- Never log sensitive data: No API keys, passwords, or PII in logs - use structured logging with redaction
- All external API calls must use circuit breakers: Prevent cascade failures from travel API outages
- All async operations require timeout: Prevent hanging requests from external services
- API responses must use standardized wrapper: Consistent response format with status, data, and error fields

**Testing Requirements**:
- **Coverage Goals**: 90%+ for business logic, 70%+ overall, 100% for external API integrations
- **Test Types**: 70% unit tests, 20% integration tests, 10% end-to-end tests
- **Framework**: pytest 7.4+ with async support
- **Mocking**: httpx-mock for API calls, pytest-mock for general mocking
- **Integration Testing**: Docker testcontainers for PostgreSQL and Redis
- **File Convention**: `test_*.py` adjacent to source files in tests/ directory
