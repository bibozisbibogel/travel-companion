### API Integration Requirements
[Source: architecture/core-workflows.md]
**Frontend-Backend Communication Flow**:
- User submits travel request via WebApp
- WebApp makes POST request to `/api/v1/trips/plan` via API Gateway
- API Gateway initiates LangGraph workflow for multi-agent planning
- WebApp receives complete itinerary response from API Gateway
- WebApp displays travel plan to user with interactive components

### Coding Standards Requirements
[Source: architecture/coding-standards.md]
**TypeScript Naming Conventions**:
- **TypeScript Interfaces**: PascalCase with I prefix (`IFlightOption`, `ITripRequest`)
- **React Components**: PascalCase (`TravelRequestForm`, `NavigationMenu`)
- **Functions and Variables**: camelCase (`handleSubmit`, `isAuthenticated`)
- **API Endpoints**: kebab-case (`/api/v1/trips/search-flights`)

**Critical Rules**:
- **TypeScript 5.3+ with strict type checking enabled**
- **ESLint + Prettier for TypeScript linting and formatting**
- **All async operations require timeout configuration**
- **API responses must use standardized wrapper format**
- **Never log sensitive data in frontend logs**

### Data Models for Frontend Integration
[Source: architecture/data-models.md]
**Frontend TypeScript Models**:
- **User Entity**: `user_id: UUID`, `email: string`, `travel_preferences: JSON`
- **Trip Entity**: `trip_id: UUID`, `user_id: UUID`, `destination: string`, `start_date: date`, `end_date: date`, `budget: number`, `status: enum`, `itinerary_data: JSON`
- **FlightOption**: `flight_id: UUID`, `airline: string`, `price: number`, `departure_time: datetime`, `arrival_time: datetime`
- **HotelOption**: `hotel_id: UUID`, `name: string`, `price_per_night: number`, `location: Point`, `rating: number`

- **Authentication Service**: JWT token management for stateless authentication

## Testing

### Testing Standards
[Source: architecture/test-strategy-and-standards.md]
- **Framework**: Vitest 1.2+ with fast, Vite-based testing and excellent TypeScript support
- **Test Files**: Create `*.test.ts` and `*.test.tsx` files adjacent to source code
- **Coverage Target**: 70%+ overall coverage, 90%+ for form validation and API integration logic
- **Test Organization**:
  - `src/__tests__/components/` - Component testing with React Testing Library
  - `src/__tests__/lib/` - API client and utilities testing
  - `src/__tests__/pages/` - Page component integration testing
- **Test Patterns**: 
  - Component testing with React Testing Library for user interactions
  - Mock external API calls with MSW (Mock Service Worker)
  - Test responsive design with viewport size testing
  - End-to-end user flows with realistic travel request scenarios

### Frontend-Specific Testing Requirements
- **Component Tests**: Test all layout components with different screen sizes and states
- **Form Validation Tests**: Test travel request form with invalid data, edge cases, and error states
- **API Integration Tests**: Mock backend API responses and test error handling scenarios
- **Accessibility Tests**: Test keyboard navigation, screen reader compatibility, and ARIA labels
- **Responsive Design Tests**: Validate layout behavior across desktop, tablet, and mobile viewports
