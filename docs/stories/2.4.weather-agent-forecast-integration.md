### External API Specifications
[Source: architecture/external-apis.md]
**OpenWeatherMap API**:
- Base URL: https://api.openweathermap.org/data/2.5/
- Authentication: API Key authentication
- Rate Limits: 60 calls/minute, 1M calls/month (free tier)
- Key Endpoints: /forecast (5-day forecast), /onecall (comprehensive weather data with alerts)
- Integration Notes: Reliable weather data with good free tier limits, essential for activity planning

**Weather API Integration Requirements**:
- All weather API calls must use circuit breaker pattern to prevent cascade failures
- Implement comprehensive rate limiting with Redis backing
- Use httpx for async API calls with proper timeout management
- All weather API responses must be cached with appropriate TTL values
- Weather forecasts should be cached for 1-3 hours depending on forecast horizon

### Coding Standards and Testing Requirements
[Source: architecture/coding-standards.md, architecture/test-strategy-and-standards.md]
**Critical Coding Standards**:
- Never log sensitive data: No API keys, passwords, or PII in logs - use structured logging with redaction
- All external API calls must use circuit breakers: Prevent cascade failures from weather API outages
- All async operations require timeout: Prevent hanging requests from external services
- API responses must use standardized wrapper: Consistent response format with status, data, and error fields

**Testing Requirements**:
- **Coverage Goals**: 90%+ for business logic, 70%+ overall, 100% for external API integrations
- **Test Types**: 70% unit tests, 20% integration tests, 10% end-to-end tests
- **Framework**: pytest 7.4+ with async support
- **Mocking**: httpx-mock for API calls, pytest-mock for general mocking
- **Integration Testing**: Docker testcontainers for PostgreSQL and Redis
- **File Convention**: `test_*.py` adjacent to source files in tests/ directory
