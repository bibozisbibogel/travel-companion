# Story 2.6: Itinerary Agent & Plan Integration

## Status
Done

## Story
**As a** user,
**I want** all my travel components integrated into a cohesive itinerary,
**so that** I have a complete, organized travel plan.

## Acceptance Criteria
1. Itinerary Agent that coordinates results from all other agents
2. Daily schedule generation with time-based activity sequencing
3. Geographic optimization to minimize travel time between activities
4. Budget calculation and allocation across all trip components
5. Conflict detection for overlapping reservations or impossible timelines
6. Itinerary export functionality with complete trip details

## Dev Notes

### Previous Story Insights
From stories 2.1-2.5, established patterns include:
- Agent coordination patterns and async interfaces from all travel agents
- Data model relationships with Trip entity as central coordinator
- Geographic Point data types for location-based calculations
- Budget and pricing decimal field patterns across all components
- External API integration patterns that need coordination
- Caching strategies for performance optimization

### Source Tree Information
[Source: docs/architecture/source-tree.md]
- **Itinerary Agent Location**: `packages/api/src/travel_companion/agents/itinerary_agent.py`
- **Trip Models**: `packages/api/src/travel_companion/models/trip.py` (extend existing)
- **Agent Dependencies**: All agents in `packages/api/src/travel_companion/agents/` (flight, hotel, activity, weather, food)
- **Services Integration**: `packages/api/src/travel_companion/services/` for caching and database operations
- **Tests Location**: `packages/api/src/tests/test_agents/test_itinerary_agent.py`

### Data Model Integration
[Source: docs/architecture/data-models.md]
- **Trip Model**: Central entity with trip_id, user_id, destination, dates, budget, status, itinerary_data JSON
- **Relationships**: Trip has many FlightOptions, HotelOptions, ActivityOptions (now including RestaurantOptions)
- **Geographic Data**: Point type for location coordinates used across all location-based models
- **Budget Integration**: Decimal fields for price calculations across all travel components

### Agent Coordination Requirements
Based on previous agent implementations (2.1-2.5):
- **Flight Agent**: Provides flight options with pricing and schedule data
- **Hotel Agent**: Provides accommodation with location and pricing  
- **Activity Agent**: Provides activities with duration, location, and pricing
- **Weather Agent**: Provides weather data for activity planning decisions
- **Food Agent**: Provides restaurant recommendations with location and pricing
- All agents must be coordinated asynchronously with timeout handling

### Technical Constraints
[Source: docs/architecture/coding-standards.md]
- **Circuit Breakers**: Agent coordination must handle individual agent failures gracefully
- **Repository Pattern**: Use repository pattern for database operations, no direct ORM calls
- **Async Operations**: All agent coordination requires timeout handling
- **Response Format**: Standardized API response wrapper with status, data, error fields
- **Geographic Calculations**: Use appropriate libraries for distance and route calculations

### Export and Integration Requirements
- **Multiple Formats**: PDF generation for printable itineraries, JSON for API consumption, iCalendar for calendar integration
- **Complete Data**: All booking details, contact information, confirmation numbers
- **Maps Integration**: Leverage Mapbox GL JS v2 for location visualization and directions
- **Real-time Updates**: Support for itinerary modifications and re-optimization

### Testing
[Source: docs/architecture/coding-standards.md]  
- **Test Framework**: pytest for Python backend testing
- **Test Organization**: Tests in packages/api/src/tests/test_agents/test_itinerary_agent.py
- **Integration Testing**: Test multi-agent coordination with mocked agent responses
- **Coverage Requirements**: Comprehensive unit tests for schedule generation, optimization algorithms, and export functionality
