# Story 3.1: Comprehensive Trip Preferences Input

## Status
Complete

## Story
**As a** user,
**I want** an intuitive, comprehensive form to specify all my travel preferences,
**so that** the system can create a personalized trip that matches my exact needs.

## Acceptance Criteria
1. Destination input with autocomplete and validation
2. Origin location input with autocomplete
3. Travel date range picker (start date, end date) with calendar interface
4. Number of travelers input with breakdown (adults, children, infants)
5. Budget input with currency type selector (USD, EUR, GBP, etc.)
6. Dietary restrictions multi-select (vegetarian, vegan, gluten-free, halal, kosher, etc.)
7. Accommodation type preferences (hotel, hostel, apartment, resort, etc.)
8. Activity types multi-select (adventure, cultural, relaxation, dining, nightlife, shopping, etc.)
9. Cuisine preferences multi-select (local, international, specific cuisines)
10. Form validation with helpful error messages and field-level feedback
11. Modern minimal design aesthetic with clean layout and clear visual hierarchy
12. Responsive design for desktop and mobile devices
13. Save draft functionality to allow users to return and complete later
14. Clear call-to-action button to generate trip itinerary

## Dev Notes

### Relevant Architecture Information
[Source: docs/architecture/tech-stack.md]
- **Frontend Framework**: Next.js 14 with React 18
- **UI Component Library**: shadcn/ui (modern minimal design system)
- **Form Management**: React Hook Form with Zod validation
- **Styling**: Tailwind CSS for responsive design
- **State Management**: React Context API or Zustand for form state

### API Integration
[Source: docs/architecture/api-design.md]
- **Trip Creation Endpoint**: `POST /api/trips` - Create new trip with preferences
- **Draft Save Endpoint**: `POST /api/trips/drafts` - Save draft preferences
- **Draft Load Endpoint**: `GET /api/trips/drafts/:userId` - Retrieve user's draft
- **Autocomplete Service**: Google Places API for destination/origin autocomplete

### Data Models
[Source: docs/architecture/data-models.md]
- **TripPreferences Model**: Contains all user input fields
  - destination: string
  - origin: string
  - start_date: date
  - end_date: date
  - travelers: {adults: number, children: number, infants: number}
  - budget: {amount: decimal, currency: string}
  - dietary_restrictions: string[]
  - accommodation_types: string[]
  - activity_types: string[]
  - cuisine_preferences: string[]

### Design System
- Use shadcn/ui components: Input, Select, Calendar, Checkbox, Button
- Color palette: Neutral grays with single accent color
- Typography: Clean sans-serif (Inter or similar)
- Spacing: Generous whitespace, 8px grid system
- Interactions: Subtle hover states, smooth transitions

### Testing Standards
[Source: docs/architecture/coding-standards.md]
- **Test Location**: `packages/frontend/src/tests/components/`
- **Testing Framework**: Jest + React Testing Library
- **Test Coverage**: Minimum 80% for form components
- **E2E Tests**: Playwright for full form submission flow

  - Issue: Frontend data format doesn't match backend API expectations
  - Backend expects: `{destination: {city, country, country_code}, requirements: {...}}`
  - Frontend sends: `{destination: "City, Country", budget: {amount, currency}, ...}`
  - Fix: Created transformTripRequestForBackend() to convert formats
  - Additional fixes for backend validation:
    - Budget converted to string (backend expects Decimal type)
    - Currency forced to uppercase (backend validation requirement)
    - Accommodation types mapped to backend enum values (hotel, apartment, hostel, resort, bed_and_breakfast, vacation_rental)
    - Country names mapped to ISO codes (FR, IT, ES, JP, etc.)
    - Destination string parsed into city/country components
  - Note: Destination search endpoint /api/v1/destinations/search returns 404
    - Already has graceful fallback to filter POPULAR_DESTINATIONS locally
